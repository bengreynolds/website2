# Future Website Recommendations

Status: living document

Purpose: track the next improvements for the portfolio site after the current content and collapsible layout pass. This document is intentionally broader than the implementation plan: it captures what is already strong, what should be refined, and what should be added later.

## What Is In Good Shape

- The site now reads as a full-scope systems portfolio instead of a narrow research-only page.
- The content has a clearer hierarchy than before.
- The hero section is stronger and no longer opens with a preselected project.
- Long content blocks are now collapsible, which reduces scroll clutter.
- The skills section has better breadth and more explicit technical detail.
- The project set is curated instead of dumping every repo.
- The visual language is more intentional than the original baseline.
- The site already has a clear foundation for future motion, 3D, and artifact-driven presentation.

## Main Risks To Watch

- The page can still drift back into repetition if future copy additions are not checked against nearby sections.
- Too many visual treatments in too many sections will create noise again.
- If 3D assets are added without a clear hierarchy, they can overpower the actual portfolio content.
- Some of the current content still reads like it belongs to research/lab work even though the broader identity is now systems engineering.
- The site needs a stronger "what can this person physically produce" signal, not just "what tools does this person know."

## Priority Roadmap

### P0 - Must Do Next

- Add real 3D renderings and animations of physical work.
- Add animation or motion samples for software interfaces when possible.
- Create a small but high-quality visual portfolio of real artifacts.
- Replace any remaining lab-centric language that weakens the full-scope systems identity.
- Keep the page from becoming visually crowded again as media is added.

### P1 - Strong Next Improvements

- Introduce a dedicated visual gallery for 3D renders, exploded views, turntables, and short UI clips.
- Add a "selected artifacts" workflow so visitors can expand into depth without being forced into it immediately.
- Build one or two stronger hero visuals that establish the range of the portfolio above the fold.
- Tighten repeated wording between hero, about, proof, and contact sections.
- Add more project-level visuals that are tied to the written description nearby.

### P2 - Valuable Later Improvements

- Add more case-study depth for the strongest projects.
- Add short process notes for build work, deployment work, and maintenance work.
- Add before/after or prototype/final comparisons where they strengthen the story.
- Add a few more project categories if they genuinely clarify the portfolio.
- Improve section pacing if new media makes the page feel too tall again.

## 3D / Motion Roadmap

### Best First Tools

1. Autodesk Fusion 360 for CAD-based stills and turntables.
2. Autodesk Inventor for assembly-based renders and motion if the source files already live there.
3. Blender for presentation scenes, animation, lighting, and compositing.

### Optional Later Tools

4. KeyShot if a fast product-render workflow becomes worth the cost.
5. Adobe Substance 3D Stager if staged product scenes become a recurring need.
6. Spline if an interactive browser-native 3D component becomes useful.

### Best Initial Asset Types

- simple turntables of finished hardware
- exploded views of mechanical assemblies
- rendered enclosures and PCB-adjacent build work
- interface mock motion for desktop applications
- short explainer loops showing workflow states
- clean "before / after" comparisons for physical or software systems

### Rules For 3D Content

- Each visual must support nearby text.
- The visual should show a real capability, not decorative motion.
- Use fewer, better assets instead of many mediocre ones.
- Keep render style consistent across the site.
- Prefer short loops, turntables, and tight camera moves before complex scenes.
- Avoid adding 3D everywhere just because it is possible.

## Content Improvements

### Hero

- Keep the full-scope systems identity explicit.
- Keep the hero short.
- Mention design, production, deployment, and support clearly.
- Avoid turning the hero into a full resume summary.

### About / Bio

- Keep the hybrid background visible.
- Keep the bio focused on how the work gets from idea to shipped system.
- Avoid repeating the same facts already shown in the hero or proof section.

### Proof / Recognition

- Keep proof factual and compact.
- Use only evidence that supports the broader systems profile.
- Do not add decorative claims or vague credibility language.

### Skills

- Keep the skills explicit and technical.
- Continue to favor concrete tools over soft categories.
- Add only skills that help explain real work or real delivery.
- Use skills to confirm fit, not to restate the same portfolio claims.

### Projects

- Keep the current curation approach.
- Ensure the strongest projects stay at the top.
- If new projects are added later, only add them if they materially improve the portfolio story.
- Avoid a "more is better" project list.

### Experience

- Keep experience entries collapsible.
- If a role is too similar to another section, trim it instead of repeating it.
- Keep the role descriptions broad enough to support industry applications without losing factual accuracy.

## Layout / UX Improvements

- Consider one stronger media strip in the hero or immediately after it.
- Consider one section that acts as a curated visual showcase instead of a long gallery.
- Reduce the number of repeated card treatments if the page starts to feel uniform.
- Keep the overall page rhythm: strong opening, selective depth, then expanding detail.
- Preserve scannability on mobile.

## Accessibility / Performance

- Keep motion respectful of reduced-motion preferences.
- Keep media load manageable.
- Avoid large uncompressed media if it hurts page speed.
- Make sure collapsible sections remain keyboard accessible.
- Ensure visuals are not the only way information is communicated.

## Current Good Stage vs. Needs Work

### Good Stage

- portfolio positioning
- broad systems identity
- collapsible content patterns
- skills breadth
- project curation
- site-wide visual language

### Needs Work

- stronger physical-production visuals
- stronger software UI visuals
- more real-world artifact presentation
- less lingering research-only language
- more differentiated section pacing
- tighter repetition control

2. Add one visual showcase area for hardware or product-style renders.
3. Add one visual showcase area for software/application motion.
4. Review copy again for repetition after new visuals are added.
5. Expand only if the site still feels too sparse after the media additions.
